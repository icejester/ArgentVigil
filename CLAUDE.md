# ArgentVigil

Silver speculative-positioning monitor with gold as comparative context. Framing is **"selling dollars, not buying metals"** — is the speculative futures crowd genuinely capitulated, or just pulling back. Not a trading system: no price targets, no prediction framing, no risk-tolerance commentary. See `SPEC.MD` (gitignored, local-only — do not assume it exists in fresh clones) for the full Stock & Flow panel spec and CATCOR feature map, and `README.md` for the user-facing feature/data-source overview.

## Architecture

Three independent layers that share one frontend. All follow **persist-on-fetch**: every upstream fetch is persisted to SQLite before anything is returned, and the frontend only ever reads from local `/db`-suffixed (or otherwise local-read) endpoints — see `SPEC.MD`'s Persist-on-Fetch Re-architecture for the full rationale (CATCOR's cross-check features assume AV's database is already the real record, not a pass-through).

1. **CoT pipeline** (`pipeline/`) — stdlib-only Python, no server. Fetches CFTC positioning data + Yahoo Finance prices, computes signals, persists an append-only archive to the shared `runtime/argentvigil.db` (via `from backend import db` — see note below), and also writes a static JSON cache (kept for standalone-CLI usability). Run manually/on a schedule, not on every request — `backend/main.py` never triggers this fetch itself. `pipeline/`'s own fetch/compute logic (`fetch.py`, `price_fetch.py`, `compute.py`) stays stdlib-only and requires no third-party packages — but it now shares `backend/db.py` for persistence rather than owning a separate SQLite file, since `backend/db.py` is itself pure `sqlite3`/`os`/`contextlib` (no `fastapi`/`httpx`) and importing it doesn't require the venv. `pipeline/run.py`'s `sys.path` bootstrap adds both `pipeline/`'s own directory and repo root so `from backend import db` resolves regardless of whether it's invoked as `python3 pipeline/run.py` (from repo root) or `python3 run.py` (from inside `pipeline/`).
2. **Exchange inventory backend** (`backend/main.py`, `backend/db.py`, `backend/mc_token.py`) — FastAPI server. Every route that talks to metalcharts.org/Sprott/FRED/Yahoo follows a fetch-and-persist function paired with a `/db` read route; a tiered background refresh (see below) keeps SQLite warm without the frontend ever calling upstream directly. Serves the built frontend as static files.
3. **CATCOR** (`backend/catcor.py`, `seed_data/catcor_events_seed.py`) — CATCOR Iteration 1's event/reaction spine, imported by `backend/main.py` the same way `backend/db.py`/`backend/mc_token.py` are (functions take an `httpx.AsyncClient` explicitly, no owned global client state). Seeds a static macro-event calendar, fetches ALFRED point-in-time actuals + ForexFactory consensus, and captures silver/gold price reactions at fixed windows around each event. See CATCOR subsection under Data flow below for the full mechanism.

`backend/`, `seed_data/`, and `runtime/` split three previously-flat-at-root concerns: app code (a real Python package, `backend/__init__.py`), manually-maintained static seed content, and gitignored generated state (the SQLite DB, CATCOR's ForexFactory cache) — the last two are not a package (no `__init__.py` in either; `seed_data/` is importable as an implicit namespace package the same way `pipeline/` already is, since uvicorn is invoked from repo root as `backend.main:app`, putting repo root on `sys.path`). Every path that used to be a bare cwd-relative string (`db.py`'s `DB_PATH`, `main.py`'s `MARKET_BALANCE_PATH` and its `frontend/dist` static mount) is now computed `__file__`-relative from each module's own location — the previous bare-relative versions only worked because `utils/dev.sh` happened to `cd` to repo root first, a latent portability bug fixed as part of this reorg.

**Where the databases live:**

- `runtime/argentvigil.db` — gitignored, owned by `backend/db.py`. **One shared database for the whole app** — every exchange-inventory, FRED, and spot-price table; CATCOR's `event_calendar`/`macro_price_reaction`/`spot_price_tick` tables; and the CoT pipeline's `cot_silver`/`cot_gold`/`cot_prices` tables (merged in from a previously-separate `pipeline/cache/cot_pipeline.db` — `pipeline/` no longer owns its own database file; `pipeline/run.py` persists via `from backend import db`, and `backend/main.py`'s `/api/cot/db` route reads the same tables directly via `db.get_silver_series()` etc., no longer through a `cot_db` alias). ## Data flow

**Standing rule (persist-on-fetch, see `SPEC.MD`):** every route that talks to an upstream source has a `_fetch_and_persist_*` function that fetches, reshapes, and writes to SQLite; the live route (unprefixed) calls that function and returns its result directly (for cases still needed, e.g. `/api/silver/delivery`), while a paired `/db`-suffixed route reads the same data straight back out of SQLite with no upstream call. The frontend calls **only** `/db`-suffixed (or `/api/cot/db`, `/api/prices/db`) routes — it never triggers a live upstream fetch itself. Something still has to call the upstream side periodically, which is what the tiered background refresh (below) is for.

- **CoT data**: `pipeline/run.py` → parses CFTC rows → persists append-only to `runtime/argentvigil.db` via `from backend import db` (`cot_silver`/`cot_gold` tables, `INSERT OR IGNORE` keyed by `report_date` — a published report is never overwritten) + upserts weekly price history (`cot_prices`, ticker+date) → **also** still writes `pipeline/cache/cot_data.json` unchanged, kept only so the pipeline remains useful as a standalone CLI tool without the backend running. The frontend no longer reads that JSON file at all: `silver_cot_tracker.jsx` calls `GET /api/cot/db`, which reads the same `cot_silver`/`cot_gold`/`cot_prices` tables directly via `backend/db.py`'s `get_silver_series`/`get_gold_series`/`get_price_series` (`backend/main.py` calls these as plain `db.*`, not through a separate `cot_db` alias — CoT's persistence was merged into the shared `db.py` module, see "Where the databases live" above) and recomputes percentiles/track-record/GSR server-side on each request (cheap — ~1,560 rows/metal) using `pipeline/compute.py`'s `compute_from_series`/`compute_signal_track_record`. `/api/cot/db` does **not** trigger a live Socrata fetch — CoT refresh stays external (`python3 pipeline/run.py`, manual or cron), a deliberate asymmetry from every other source's `/refresh`+`/db` pattern, since a 15-year Socrata pull is too slow to trigger from an HTTP request. The CoT panel no longer renders a "Macro Watchlist" section — that manually-edited UI was removed, and `/api/cot/db`'s response drops `macro_watchlist` entirely (not ported to SQLite).
- **Exchange inventory (silver + gold)**: each of COMEX/SHFE/PSLV history, depositories, leverage/OI, delivery notices, and SHFE warehouses has a `_fetch_and_persist_*` function in `main.py` that fetches from metalcharts.org (authenticated via `mc_token.py`) or Sprott, persists to SQLite, and a paired `/db` route reads it back (e.g. `/api/silver/db/depositories`, `/api/silver/db/leverage`, `/api/silver/db/delivery`, `/api/shfe/db/warehouses`, `/api/pslv/db`). Gold has a full parallel route set to silver (`/api/gold/history`, `/api/gold/db/history`, `/api/gold/depositories`, `/api/gold/db/depositories`, `/api/gold/leverage`, `/api/gold/db/leverage`) using `symbol=XAU` and 100oz contracts (vs. silver's `XAG`/5,000oz) — confirmed live against metalcharts.org before building, though only the leverage route currently has a frontend consumer (gold COMEX history/depositories are fetched and persisted every slow-tier cycle but nothing in `comex_inventory.jsx` renders them yet — a known no-op, not a bug). `/api/silver/db/leverage` and `/api/gold/db/leverage` power the CoT panel's per-metal Paper Leverage cards, each also showing a spot price via `/api/prices/db` (backed by a new `spot_price_snapshot` table — kept separate from `fred_observations`' `XAG_CLOSE`/`XAU_CLOSE` daily-close keys since the live spot quote carries a `change_pct_24h` field the generic FRED-style table doesn't have room for).
- **Tiered background refresh** (`main.py`'s `lifespan`): two independent `asyncio` loops keep SQLite warm without the frontend ever calling upstream. **Fast tier** = spot prices only (`/api/prices`'s fetch+persist logic) — the one genuinely intraday-moving figure. **Slow tier** = the other 10 sources (silver/gold COMEX history, depositories ×2, leverage ×2, delivery notices, SHFE history, SHFE warehouses, PSLV) — all move at most daily upstream. Both tiers run once, unconditionally, at server startup (so the app has data immediately even with both toggles off), then default to **disabled** — each tier only repeats on its interval if explicitly turned on. State (interval + enabled flags per tier) is in-memory only, read/written via `GET`/`POST /api/refresh/settings` (body: `{"fast_interval_s", "slow_interval_s", "fast_enabled", "slow_enabled"}`); `POST /api/refresh/force` runs both tiers once immediately regardless of their enabled state and returns real per-tier `{"succeeded", "failed", "errors"}` counts (not a blind `{"success": true}`) so the frontend can tell a genuine failure — e.g. no network — from a real update, and only tells other panels to re-fetch (`refresh_controls.jsx`'s `FORCE_REFRESH_EVENT`, listened for by `comex_inventory.jsx` and `PaperLeveragePanel`) when at least one source actually succeeded. CoT is intentionally excluded from force-update — it stays pipeline-only.
- **Market balance**: `main.py` reads `silver_market_balance.json` directly (no DB, no external fetch) and computes derived metrics (5yr cumulative, runway years, staleness) on each request. Demand Composition Over Time is now a separate exported component (`DemandCompositionPanel` in `market_balance.jsx`) reading the same balance data, so it can be independently collapsible in the UI.
- **Money supply**: frontend calls `/api/fred/money-supply/refresh` (fetches M2SL, WALCL, CPIAUCSL from FRED, upserts SQLite `fred_observations`) and `/api/fred/money-supply/db?window=2y|5y|10y|20y` (reads SQLite, computes YoY + CPI-derived purchasing-power index server-side, returns trillions/percent/index — never raw FRED units). Requires `FRED_API_KEY` in the environment (`.env`, loaded via `python-dotenv`, or shell-exported — either works, see `main.py`'s `load_dotenv()`); `refresh` raises a clean `500` if unset rather than failing silently.
- **Metal price history (for Money Supply's purchasing-power comparison)**: `/api/metals/prices/refresh` fetches DAILY closes from Yahoo Finance (`SI=F`/`GC=F`, httpx directly in `main.py` — not `pipeline/price_fetch.py`, which is a separate stdlib-only weekly-interval helper for the CoT pipeline), resamples to one row per calendar month (last trading day on/before month-end, avoiding Yahoo's `1mo` bucket which includes the current in-progress month), and upserts into the same generic `fred_observations` table under `XAG_CLOSE`/`XAU_CLOSE` series_id keys. `/api/metals/prices/db?window=...` reads that back and returns both raw price and an index-to-100-at-window-start value per point. The frontend then does its own further client-side rebasing against a user-selected baseline (see money_supply.jsx above) — the backend only supplies window-start-indexed values, not baseline-relative ones.
- **CATCOR (event/reaction spine)**: `main.py`'s `lifespan` runs `catcor.seed_events()` (loads `catcor_events_seed.py`'s static FOMC/CPI/NFP list, upserts into `event_calendar` for the seeded 90-day-back/60-day-forward window, `event_id` deterministically `f"{event_type}_{date}"` so re-seeding is idempotent) → `catcor.backfill_intraday_ticks()` + `catcor.backfill_daily_closes()` (Yahoo Finance, see below) → `catcor.fetch_and_persist_consensus()` (ForexFactory) → `catcor.fetch_and_persist_actuals()` (ALFRED) → `catcor.backfill_reactions()`, all best-effort/independently try-excepted so one failure doesn't block the rest. Three tables in `argentvigil.db`: `event_calendar` (event_id PK, includes `source_tier` — currently always `"government"`, see catcor_events_seed.py above), `macro_price_reaction` (PK event_id+metal+window, one row per captured T-30m/T+5m/T+30m/T+2h price reaction), and `spot_price_tick` (PK series_id+ts, **append-only** `INSERT OR IGNORE` — fixes a real gap `spot_price_snapshot` has: that table is one row per calendar day, overwritten on every fast-tier tick, so it has no true intraday series to snapshot from; `spot_price_tick` does, and `main.py`'s `_fetch_and_persist_prices` now writes to both tables on every fast-tier poll).
  - **ALFRED actuals**: `_fetch_alfred_change` fetches two consecutive vintage prints for `CPIAUCSL`/`PAYEMS` as of the day after `scheduled_time` and diffs them — a raw ALFRED level is NOT comparable to ForexFactory's consensus (a period-over-period change), a real bug caught during build. NFP's diff is a raw count (thousands of persons, matching PAYEMS's native unit); CPI's is a % m/m change. `surprise_delta = actual - consensus`, computed whenever both are known (by either fetch running first — order doesn't matter, each recomputes `surprise_delta` if the other value is already present).
  - **Consensus**: `fetch_and_persist_consensus` matches `event_calendar` rows against ForexFactory's free, no-key `ff_calendar_thisweek.json` feed (title + same-instant match, "Non-Farm Employment Change" specifically — not "ADP Non-Farm Employment Change", a different report) by `event_type`. Only a "this calendar week" (Sun-Sat) variant exists — nextweek/lastweek both 404 — so consensus can only ever be captured for events within roughly ±6 days of "now," not retroactively for older backfilled events; this is a permanent, disclosed gap, not a bug. The feed is fetched **at most once per calendar week** and cached to `runtime/forexfactory_thisweek.json` (keyed by that week's Sunday date) — confirmed live that repeat hits trip the endpoint's rate limit (429) and a wider community report says it can lock out an IP entirely, so `catcor.py` never re-fetches within the same week regardless of how often `fetch_and_persist_consensus` is called. A dedicated `_consensus_tier_loop` (`main.py`, 30min interval) calls it periodically — alongside `fetch_and_persist_actuals` (ALFRED shares this loop rather than getting its own, since checking more often than consensus does buys nothing) — so an event that enters the current week after startup is still caught; almost every one of those calls is a pure cache read, not a network hit. ForexFactory's `"K"`-suffixed NFP forecasts (e.g. `"114K"`) must be scaled to PAYEMS's native thousands unit (÷1000, not the naive ×1000) — a second real unit bug caught during build alongside the ALFRED-level-vs-change one above.
  - **Snapshot capture** (`catcor.capture_snapshot`): idempotent by construction — skips entirely if a `macro_price_reaction` row already exists for the given `(event_id, metal, window)`, so repeated calls (a normal restart, or a restart after any length of downtime) are cheap; only genuinely-missing windows do real work. `due_snapshots()` finds `(event_id, window)` pairs whose boundary has passed but aren't fully captured yet — polled every 60s by `main.py`'s `_event_tier_loop` (always-on, not gated by an enabled flag like fast/slow tier, since a missed snapshot window is permanent data loss, unlike fast/slow tier data which is always re-fetchable). Price lookup: `db.get_ticks_near` finds the nearest `spot_price_tick` row within a tolerance window; if none exists, falls back to `catcor.backfill_daily_closes`'s dedicated `XAG_DAILY_CLOSE`/`XAU_DAILY_CLOSE` `fred_observations` keys (NOT the existing `XAG_CLOSE`/`XAU_CLOSE` keys — those are month-end resampled for Money Supply and have no daily granularity at all, a real bug caught during build when the fallback silently returned `NULL` against the live DB). `backfill_intraday_ticks` pulls Yahoo Finance 5-minute bars (`interval=5m`, capped at ~72 days back by Yahoo; 1-minute interval is capped at only 8 days, too short to be useful) — so roughly 72 of a 90-day backfill window get real intraday-precision reactions, and the remaining ~18 oldest days fall back to the coarser same-day daily-close approximation.
  - **Routes**: `GET /api/catcor/events/db` (upcoming events), `GET /api/catcor/reactions/db` (full event×metal×window join for the frontend's two charts), `POST /api/catcor/refresh` (manual re-seed + backfill + re-fetch — returns a clean `500` if `FRED_API_KEY` is unset, same pattern as `fred_money_supply_refresh`).

## Conventions

- **Backend**: stdlib `sqlite3` (no ORM), `httpx.AsyncClient` for outbound calls (one shared client via FastAPI lifespan), routes are thin — fetch, reshape, upsert, return `{"success": bool, "data": ...}`. HTTP errors from upstream become `HTTPException(502, str(e))`.
- **Pipeline**: intentionally stdlib-only, no third-party deps — keep it that way (it's meant to run standalone without the venv). Config constants belong in `pipeline/config.py`, not scattered inline.
- **Frontend**: React 19 + Vite 5 + Recharts, no state library, no router (single page, tab-like sections via `App.jsx`). Chart colors come from `palette.js`, not ad hoc hex values.
- **Units**: COMEX figures are troy oz; SHFE is kg natively, converted via `1 kg = 32.1507 oz` (constant appears in both `main.py` and any frontend display — keep consistent). COMEX open interest contracts are 5,000 oz each.
- **Nulls over zeros**: metalcharts.org sometimes reports `0` to mean "not reported that day" — `_parse_aggregate_row` in `main.py` converts those to `None` so charts gap instead of showing false dips. Apply the same treatment to any new fields with this failure mode.
- **AV Voice Rules** (from SPEC.MD, applies repo-wide to any user-facing copy): no "alarming"/"critical" framing on deficit trends, always show the ±20% uncertainty qualifier on above-ground stock figures, label runway/track-record figures explicitly as historical description or context — never as prediction.
- **Collapsible sections**: the established UI pattern for any collapsible pane is native `<details className="collapsible-pane"><summary className="collapsible-pane-title">…</summary><div className="collapsible-pane-body">…</div></details>` (CSS in `index.css`) — nest freely (e.g. COMEX/SHFE sections each contain their own nested sub-collapsibles). Prefer this over a `useState` show/hide toggle for new collapsible UI unless more complex interaction state is needed.
- **Custom legends over Recharts' built-in `<Legend>`**: chart legends in this codebase are hand-rolled (`.comex-legend-list`/`.comex-legend-item`/`.comex-legend-swatch` classes), one entry per line with an inline plain-English definition — not Recharts' default legend component. Some legend rows are also clickable (e.g. toggle a line's visibility, or select it as a baseline) via a wrapping `<button className="comex-legend-item legend-btn-row">`.
- **Custom scatter-point shapes for cross-chart consistency**: CATCOR's two charts and its legend all use the same diamond shape per event type (`.comex-legend-swatch--diamond` for the legend; a shared `makeLinkedShape` renderer for the charts themselves) so an event type reads identically everywhere rather than circles in one chart and a line-swatch in the legend.

## Running it

```bash
bash utils/dev.sh          # everything (venv bootstrap + backend :8000 + frontend :5173)
python3 pipeline/run.py    # CoT pipeline only, no server needed
```

Run the pipeline at least once before the frontend, since the CoT tab reads from `/api/cot/db`, which reads `cot_silver`/`cot_gold` from `runtime/argentvigil.db` — empty tables make that route return a `500` ("No CoT data persisted yet. Run pipeline/run.py first.").

The exchange-inventory/FRED/spot-price data populates itself on first backend startup regardless (a one-time unconditional refresh runs in `main.py`'s `lifespan`) — the tiered background refresh only *repeats* on a schedule if explicitly enabled via the "Configuration" panel in the UI (both tiers default off) or via `POST /api/refresh/force`. CATCOR's event calendar and reaction backfill also run automatically on every startup (`catcor.seed_events()` etc., see Data flow's CATCOR subsection) — no manual trigger needed, though `/api/catcor/refresh` exists for an on-demand re-run. ALFRED calls need `FRED_API_KEY` set in whatever shell launches the backend (same requirement as Money Supply, above) — without it, the event calendar and price reactions still populate, but `actual_value`/`surprise_delta` stay `NULL`.

**Always run Python through `.venv`, never bare `python3`.** `bash utils/dev.sh` creates
`.venv` on first run and installs `requirements.txt` into it (`fastapi`, `uvicorn`, `httpx`,
`python-dotenv` — none of these are on system Python). Before running any Python command
against this repo — import checks, one-off scripts, `python -c "..."` sanity tests — run
`source .venv/bin/activate` first, or invoke `.venv/bin/python` directly. `pipeline/` is the one
exception: it's intentionally stdlib-only by design (see Conventions above) and *can* run under
bare `python3`, but `main.py`/`db.py`/anything importing `fastapi`, `httpx`, or `dotenv` cannot.
