## Standing architectural rules (apply to every tab below)

- **Persist-on-fetch**: every route that talks to an upstream source has a `_fetch_and_persist_*` function that fetches, reshapes, and writes to SQLite; a paired `/db`-suffixed route reads the same data straight back out with no upstream call. **The frontend calls only `/db`-suffixed (or `/api/cot/db`, `/api/prices/db`) routes ŌĆö it never triggers a live upstream fetch itself.** Rationale: CATCOR's and Delivery Behavior's cross-check features assume AV's database is already the real record, not a pass-through ŌĆö a route that fetched live on every request would let two simultaneous reads observe two different upstream states, breaking that assumption. (Formerly documented in the retired `SPEC.MD`'s "Persist-on-Fetch Re-architecture" section ŌĆö see the note on `SPEC.MD` above; this is now the canonical statement of the rule.)
- **One shared database**: `runtime/argentvigil.db` (gitignored), owned by `backend/db.py`, `DB_PATH` computed `__file__`-relative (two directories up from `backend/db.py`, into `runtime/`). Every tab's tables live here ŌĆö there is no per-tab or per-layer database file. `runtime/` contains nothing else (an earlier flat-file cache for CATCOR's ForexFactory feed was migrated into a table ŌĆö see CATCOR section below).
- **Backend package layout**: `backend/` is a real package (`backend/__init__.py`, empty, enables `from . import db` and `uvicorn backend.main:app`). `seed_data/` (manually-maintained static content) and `runtime/` (gitignored generated state) are namespace packages / non-packages sitting alongside it, not nested inside it.
- **Tiered background refresh** (`main.py`'s `lifespan`): two independent `asyncio` loops keep SQLite warm without the frontend ever calling upstream. **Fast tier** = spot prices only (the one genuinely intraday-moving figure). **Slow tier** = the other ~10 exchange-inventory sources (all move at most daily upstream). Both run once, unconditionally, at server startup, then default to **disabled** ŌĆö each tier only repeats on its interval if explicitly turned on via `GET`/`POST /api/refresh/settings`. `POST /api/refresh/force` runs both tiers once immediately regardless of enabled state, returns real per-tier `{"succeeded","failed","errors"}` counts, and ŌĆö only if at least one source actually succeeded ŌĆö dispatches a `FORCE_REFRESH_EVENT` window `CustomEvent` that Inventory/CoT/CATCOR panels listen for to re-read their `/db` data immediately. CoT and Research are excluded from force-update (CoT stays pipeline-only; Research is on-demand chat, nothing to force-refresh).
- **Nulls over zeros**: metalcharts.org sometimes reports `0` to mean "not reported that day" ŌĆö `_parse_aggregate_row` in `main.py` converts those to `None` so charts gap instead of showing false dips. Apply the same treatment to any new fields with this failure mode.
- **Stale spot feed over weekends/closures**: `/api/prices`'s underlying `twelvedata-ws` feed doesn't pause when COMEX/LBMA silver/gold trading is actually closed ŌĆö confirmed live over a weekend that it kept returning a top-level `isStale: true` flag (sibling of `data`, not per-metal) alongside a `cacheAge` in the hours, while still stamping each entry with a live-looking `timestamp` and slowly drifting the price via rounding/re-sampling jitter on metalcharts.org's own end. `_fetch_and_persist_prices` (`main.py`) checks `isStale` **and** `date.today().weekday() >= 5` (real Sat/Sun) before skipping persistence ŌĆö no new `spot_price_snapshot`/`spot_price_tick` row is written on a real weekend, so `PriceHistoryChart` goes flat at the last real print instead of showing fake weekend movement that never actually traded. On a weekday, `isStale=true` is persisted anyway rather than skipped ŌĆö confirmed live that `isStale` can also fire on a weekday with `cacheAge` in the **months** (metalcharts.org's own upstream feed stuck, not a market closure); skipping indefinitely in that case would silently flatline the chart forever with no visible signal anything's wrong, so a stuck weekday feed is deliberately allowed to surface directly in the chart instead. Apply the same check to any other route that reads `/api/prices`' response.
- **The Data tab MUST be updated whenever app data changes ŌĆö this is a strict requirement, not a nice-to-have.** As of datasources-spec.md's Story #1 split, this now spans two homes by kind of change: operational metadata (cadence, rate limit, `requires_env`, table ownership, affinity group) lives in **`backend/sources.py`** ŌĆö any new/changed source's fetch cadence or rate-limit posture must land a matching `SourceDefinition`/`CadenceSpec`/`RateLimitSpec` edit there; editorial prose (per-field descriptions, curl examples, origin text) stays hand-written in **`frontend/src/data_editorial.js`** (`frontend/src/data_panel.jsx` renders both, joined by `sourceKeys`). Any change that adds/removes/renames a SQLite table or column, adds/removes/changes an upstream source or its fetch shape, or changes a source's fetch cadence/rate-limit posture must land a matching edit to the relevant one (or both) of these files in the same change. Treat a data-shape change without a Data-tab update as incomplete, the same as a migration without a schema update.
- **Pinned default tab**: a ­¤ōī icon on each of `App.jsx`'s six nav buttons lets the user designate which tab opens by default on load, instead of always defaulting to CoT. Persisted server-side (`ui_settings` table, single-row upsert ŌĆö same convention as `pipeline_runs` ŌĆö via `GET`/`POST /api/ui/pinned-section`), **not** `localStorage` (the app has no client-only persistence anywhere; a shared setting like this belongs in the same SQLite database as everything else). `App.jsx` fetches the pinned section on mount and opens it if valid; falls back to `"cot"` if nothing's pinned or the fetch fails.

## Dev Notes (cross-spec, pre-feature findings)

Investigation results, environment quirks, and open threads that don't
belong to a shipped tab yet. Promote to a tab section (with a real spec)
once something here becomes a real feature; delete the note once promoted

- **Yahoo Finance chart API query-shape sensitivity** (confirmed working
  as of 2026-07, see `squeeze-context-spec.md` Story #1): an earlier,
  isolated test of `interval=1d&range=5d` against a deferred-month
  contract symbol returned HTTP 429, raising a concern that this query
  shape or symbol type was specifically rate-gated. A follow-up
  investigation re-tested it directly ŌĆö six back-to-back requests (no
  delay) against real deferred-month symbols (`SIU26.CMX`, `SIZ26.CMX`,
  `GCQ26.CMX`, `GCZ26.CMX`, plus `SI=F`/`GC=F` controls), using that exact
  query shape, all returned HTTP 200 with distinct real prices per
  contract month and a full year of daily history available. No 429s
  observed. The original 429 is now believed to have been a transient/
  IP-cooldown artifact from earlier rapid-fire testing, not a query-shape
  or symbol-type block ŌĆö consistent with this endpoint's broader usage
  elsewhere in this codebase (`catcor.py`'s startup backfill already hits
  this same endpoint/shape against `SI=F`/`GC=F`, `main.py`'s
  `_fetch_yahoo_daily_closes` does too on-demand, with no documented 429
  anywhere, unlike ForexFactory's confirmed one ŌĆö see `catcor.py`'s own
  rate-limit comment). This endpoint has no published quota and no
  rate-limit response headers (confirmed live) ŌĆö any throttling posture
  has to stay conservative-by-design, not measured. `futures_curve_spread`
  ingestion (`main.py`'s `_fetch_and_persist_curve_spread`) uses a single
  retry with short backoff on 429/5xx as a defensive margin, not because a
  real recurring lockout has been observed.
- **COMEX contract-month liquidity does not match the textbook delivery
  cycle** (confirmed live, 2026-07, during the same investigation): Yahoo
  returns a price for every calendar-month futures symbol, including
  thin/illiquid ones. Silver's real near-term depth was concentrated in
  Sep (`SIU26.CMX`, vol=14,445) and Dec (`SIZ26.CMX`, vol=1,123) only,
  while Jul (`SIN26.CMX`, textbook "active" month) showed just vol=5; gold's
  real depth was Aug (`GCQ26.CMX`, vol=22,196) and Dec (`GCZ26.CMX`,
  vol=1,429), while Oct/Jul/Sep/Nov were all under 200. A hand-maintained
  "active delivery months" list was tried first for `futures_curve_spread`'s
  front/next-month resolution and abandoned once this was confirmed ŌĆö see
  Tab: CoT's Squeeze Context subsection for the live-volume-ranking
  approach used instead. Relevant beyond this feature: any future work
  that needs "which COMEX contract month is currently liquid" should
  re-derive it live rather than trust a static list, including this note's
  own numbers, which are a snapshot, not a standing rule.

## Tab: CoT

**Frontend**: `frontend/src/silver_cot_tracker.jsx`. **Backend**: `pipeline/` (fetch + compute) + `backend/main.py`'s `/api/cot/db` route + `backend/delivery_behavior.py`'s category-composition compute (see below ŌĆö this content is CoT-derived, not Stock & Flow, so it lives in this tab despite being part of the Delivery Behavior module).

### CoT: what it does

Fetches CFTC Commitment of Traders data for COMEX Silver (084691) and COMEX Gold (088691), computes a normalized positioning metric (`net_long_pct_oi` = (non-commercial longs ŌłÆ non-commercial shorts) ├Ę total open interest ├Ś 100), and ranks it against rolling 2-year and 5-year historical windows. Normalizing by OI makes readings from 2011 and 2024 directly comparable.

**Signal zones** (configurable in `pipeline/config.py`): Ōēź90th percentile ŌåÆ specs crowded long (caution); Ōēż10th percentile ŌåÆ specs capitulated (back-up-the-truck zone); between ŌåÆ normal range, no signal. Both windows computed side-by-side; the UI flags disagreement explicitly.

### CoT: data flow

`pipeline/run.py` ŌåÆ parses CFTC rows (via `pipeline/fetch.py`, stdlib-only Socrata client) ŌåÆ persists append-only to `cot_silver`/`cot_gold` (`INSERT OR IGNORE` keyed by `report_date` ŌĆö a published report is never overwritten) + upserts `cot_prices` (ticker+date, weekly, via `pipeline/price_fetch.py`, Yahoo Finance) ŌåÆ stamps `pipeline_runs` (single-row, `id=1`) with completion time, unconditionally, even if no new CoT rows landed ŌåÆ also writes `pipeline/cache/cot_data.json` unchanged (standalone-CLI usability only; frontend never reads it). Persistence goes through `from backend import db` (shared `backend/db.py`, itself pure `sqlite3`/`os`/`contextlib`, no `fastapi`/`httpx`, so importing it doesn't need the venv) ŌĆö `pipeline/` no longer owns a separate database file.

`GET /api/cot/db` reads `cot_silver`/`cot_gold`/`cot_prices` directly via `db.get_silver_series`/`get_gold_series`/`get_price_series` and recomputes percentiles/track-record/GSR server-side on each request (cheap, ~1,560 rows/metal) using `pipeline/compute.py`'s `compute_from_series`/`compute_signal_track_record` (`compute_from_series` is an additive twin of `parse_and_compute` that takes already-normalized SQLite rows instead of raw CFTC rows). **`/api/cot/db` never triggers a live Socrata fetch** ŌĆö CoT refresh stays external (`python3 pipeline/run.py`, manual or cron), a deliberate asymmetry from every other tab's `/refresh`+`/db` pattern, since a 15-year Socrata pull is too slow to trigger from an HTTP request. `POST /api/refresh/force` does not touch CoT or `pipeline_runs` either.

As of the Data Health work (see Data tab below), `cot_pipeline` also has its own registry entry (`_refresh_cot_pipeline` in `main.py`) ŌĆö a rate-limit gate (skips, `last_attempt_status="skipped"`, if the latest persisted report is under 7 days old) wrapping `asyncio.to_thread(pipeline_run.run_pipeline_once)`. This is the first in-app trigger CoT has ever had, reachable via the Data tab's per-source "Re-run now" button; it does not change the fact that `/api/cot/db` itself stays read-only.

### Squeeze Context (`squeeze-context-spec.md`)

Pairs COT positioning extremity with a futures curve spread, plus a small hand-maintained historical case log ŌĆö "was the specs crowd genuinely stretched, or is this just the front-vs-next-month curve shape," each series carrying its own explicit as-of date, never implied simultaneity. Not a composite score, not an alerting system, not a new tab. Grew out of a conversation about the January 2026 silver spike (>$120/oz), where curve backwardation and stretched Managed Money positioning both preceded the move.

- **Futures curve spread** (`futures_curve_spread` table, `main.py`'s `_fetch_and_persist_curve_spread`, slow tier, `GET /api/curve-spread/db?metal=`) ŌĆö front-month vs. next-month COMEX settlement spread, both metals, daily. `curve_spread_pct = (next_month_price - front_month_price) / front_month_price`; positive = contango, negative = backwardation; NULL (not 0) on any day either leg has no real price. Sourced from Yahoo Finance's chart API against real deferred-month contract symbols (e.g. `SIU26.CMX`) ŌĆö see the Dev Notes section above for the rate-limit and contract-liquidity investigation that unblocked this.
  - **Front/next-month resolution is liquidity-ranked PER HISTORICAL DATE, not once for "today" and backfilled** ŌĆö this went through two real bugs, both caught by inspecting the persisted data after initial ship rather than trusting the mechanism on paper. **Bug 1**: the first version ranked liquidity once for today's date, then fetched each of those two symbols' entire trailing-year history and stored it all under that single label. Since which contract is genuinely front/next changes over time (confirmed live: during the real January 2026 squeeze, the actually-liquid front month was `SIN26.CMX`/Jul26, not `SIU26.CMX`/Sep26 ŌĆö what a "rank once for today" approach would have wrongly applied retroactively), this produced a spread series that looked suspiciously flat (0.71%-1.13% for the entire year, barely reacting to a real >80% spot move) ŌĆö not because the underlying Yahoo prices were fake (they're real, independently-traded, with real daily volume ŌĆö confirmed via each symbol's `longName`/distinct 52-week-high-low/distinct daily volume, e.g. `SIU26.CMX` had genuine 14,445-volume days and `SIN26.CMX` had genuine 2,800-9,700-volume days during the squeeze itself), but because the LABEL was wrong for most of the backfill window. Fixed by fetching each candidate symbol's full daily (close, volume) history once (`_fetch_yahoo_contract_daily` now returns per-day volume from the response's own `indicators.quote[0].volume` array, not just `meta.regularMarketVolume`'s today-only snapshot), then re-ranking front/next independently for every date using that date's own real volume. **Bug 2**, caught immediately after fixing Bug 1: pure per-date volume ranking (highest volume = front, second-highest = next) could pair an about-to-expire contract's trailing volume against a genuinely newer front month and call the chronologically EARLIER contract "next" ŌĆö e.g. early Dec 2025 briefly ranked `SIZ25.CMX`/Dec25 (winding down, still had residual volume) as second-highest against `SIN26.CMX`/Jul26 (the real front month by then), producing a spurious ~-2.7% "negative spread" that was a backwards pairing, not real backwardation. Fixed with `_delivery_sort_key` (parses year/month from the symbol) ŌĆö "next" must be the highest-volume REMAINING candidate whose delivery month is strictly later than front's, not just the second-highest by volume regardless of month order. **After both fixes, real data across the full trailing year (2025-05 through today) never goes negative** ŌĆö range is roughly +0.6% to +4.6%, tightest during the January 2026 squeeze itself (~0.6-0.7%, `SIN26.CMX`/`SIU26.CMX`) and widest for the widest month-gap pair encountered (`SIZ25.CMX`/`SIZ26.CMX`, a full year apart, naturally carries more cost-of-carry). This is treated as a real finding, not a remaining bug ŌĆö the front/next spread specifically didn't show backwardation for this particular event; that may mean physical stress concentrated more at spot-vs-front-month than between two already-deferred paper contracts (see the spot-vs-front-month series below, which is forward-looking-only and can't retroactively confirm this).
  - `_candidate_contract_symbols` sweeps both directions from today (`CURVE_SPREAD_CANDIDATE_MONTHS_AHEAD`=8, `CURVE_SPREAD_CANDIDATE_MONTHS_BEHIND`=14) ŌĆö the "behind" sweep is what Bug 1's fix needed: enough trailing candidate months that a per-date backfill ranking has every symbol that could have genuinely been front/next at any date in the ~370-day fetch window, not just symbols visible from today's vantage point.
  - A hand-maintained "active delivery months" list (e.g. the textbook silver Mar/May/Jul/Sep/Dec cycle) was tried first for month *selection* and abandoned even before the two bugs above, once live testing showed COMEX's real near-term liquidity doesn't match that textbook cycle (see Dev Notes) ŌĆö mirrors the same reasoning `delivery_behavior.py`'s own module docstring gives for computing FND/LTD per-month on demand rather than off a small fixed list.
  - Single retry with short backoff on 429/5xx per request, then skip-and-log without failing the rest of the slow tier.

### Money Supply: data flow

`GET /api/fred/money-supply/refresh` fetches M2SL/WALCL/CPIAUCSL/WRESBAL/RRPONTSYD/WSHOTSL/WSHOMCB/WLCFLPCL from FRED, upserts `fred_observations` (generic `(series_id, date, value)` store, not FRED-exclusive ŌĆö also holds Yahoo-sourced metal closes, see below). `GET /api/fred/money-supply/db?window=2y|5y|10y|20y` reads it back, computes YoY + CPI-derived purchasing-power index server-side, returns trillions/percent/index (never raw FRED units). Requires `FRED_API_KEY` in the environment (`.env` via `python-dotenv`, or shell-exported); `refresh` raises a clean `500` if unset.

**Unit note, confirmed live against FRED's `/fred/series` metadata**: the 5 Fed Balance Sheet Composition series do **not** all share a native unit despite all being simple weekly series ŌĆö WRESBAL/WSHOTSL/WSHOMCB/WLCFLPCL report in millions of USD (same convention as WALCL), RRPONTSYD reports in billions (same convention as M2SL). `/api/fred/money-supply/db` converts each with its own correct divisor (`_millions_to_trillions` helper for the four millions-denominated series); assuming a shared divisor was a real bug caught during build (an early version divided WRESBAL by 1,000 same as RRPONTSYD, producing a nonsensical ~$2,951T bank-reserves figure).

**Metal price history** (for the purchasing-power chart): `/api/metals/prices/refresh` fetches DAILY closes from Yahoo Finance (`SI=F`/`GC=F`, httpx directly in `main.py` ŌĆö a separate code path from `pipeline/price_fetch.py`, which is a stdlib-only weekly-interval helper for the CoT pipeline specifically), resamples to one row per calendar month (last trading day on/before month-end, avoiding Yahoo's `1mo` bucket which includes the current in-progress month), upserts into `fred_observations` under `XAG_CLOSE`/`XAU_CLOSE` keys. `/api/metals/prices/db?window=...` returns raw price + index-to-100-at-window-start; the frontend does its own further client-side rebasing against the user-selected baseline.

## Tab: CATCOR (Catalyst Correlation)

**Frontend**: `frontend/src/catcor_panel.jsx`. **Backend**: `backend/catcor.py`, `seed_data/catcor_events_seed.py`.

### CATCOR: what it does

CATCOR Iteration 1's event/reaction spine ŌĆö answers "did this catalyst actually move the metal" with observed data only, no interpretation layer. Tracks a macro-event calendar (FOMC, CPI, NFP), sources actual/consensus values from ALFRED/ForexFactory, captures silver/gold price reactions at fixed windows (T-30min/T+5min/T+30min/T+2hr) around each event.

### CATCOR: panel contents

Two Recharts scatter charts sharing a metal (XAG/XAU) toggle:

- **Catalyst Timeline** (top) ŌĆö every captured catalyst by real calendar date, 1M/3M/6M/1Y lookback (fixed 30-day lookahead). Includes events with no known `surprise_magnitude` yet; known-future events render at reduced opacity as placeholders (never a real 0% reaction).
- **Surprise Magnitude vs. Price Reaction** (below, own T-30m/T+5m/T+30m/T+2h window toggle) ŌĆö only plots events where a real `surprise_magnitude` exists.

Government-seeded events (FOMC/CPI/NFP) render as diamonds; Research-promoted (`observed`) events render as **circles** in their own amber color ŌĆö event-type colored via `CATCOR_EVENT_COLORS` either way. Hovering a point in either chart doubles the size of that same event's point in the other (shared `hoveredEventId` state); clicking an `observed` point hotlinks to its originating Research session (see Tab: Research's "Research Ōåö CATCOR integration" for the full mechanism ŌĆö promote/demote, the hotlink wiring, and clickable legend rows all live there since they're driven by the Research tab's disposition flow, not CATCOR's own data).

### CATCOR: data flow

`main.py`'s `lifespan` runs, in order (all best-effort/independently try-excepted): `catcor.seed_events()` (loads `catcor_events_seed.py`'s static FOMC/CPI/NFP list, upserts into `event_calendar` for a 90-day-back/60-day-forward window, `event_id = f"{event_type}_{date}"` so re-seeding is idempotent; each event carries `source_tier`, currently always `"government"`) ŌåÆ `catcor.backfill_intraday_ticks()` + `backfill_daily_closes()` (Yahoo Finance) ŌåÆ `fetch_and_persist_consensus()` (ForexFactory) ŌåÆ `fetch_and_persist_actuals()` (ALFRED) ŌåÆ `backfill_reactions()`.

Three core tables: `event_calendar` (event_id PK), `macro_price_reaction` (PK event_id+metal+window), `spot_price_tick` (PK series_id+ts, **append-only** ŌĆö fixes a gap `spot_price_snapshot` has: one row per calendar day, overwritten every fast-tier tick, no true intraday series; `_fetch_and_persist_prices` writes to both tables on every fast-tier poll).

**`spot_price_tick`'s series_id families ŌĆö a real collision, since fixed**: `"XAG"`/`"XAU"` are exclusively `main.py`'s real metalcharts.org spot ticks (60s cadence, power the CoT tab's `PriceHistoryChart`). `"XAG_FUTURES"`/`"XAU_FUTURES"` (`FUTURES_SERIES_ID` in `catcor.py`) are Yahoo SI=F/GC=F futures bars from `backfill_intraday_ticks`, a genuinely different instrument, used only for CATCOR's own event-reaction snapshot capture below. These two used to share the bare `"XAG"`/`"XAU"` keys ŌĆö futures prints running systematically ~0.3ŌĆō0.6 higher than spot, interleaved on the same series, produced a real sawtooth artifact in the CoT price chart. Existing historical rows were migrated (re-keyed by a 5-minute-grid timestamp signature ŌĆö Yahoo's bars land exactly on `:00`-second, no-microsecond, 5-min-multiple timestamps with zero false positives found against the real spot series) rather than discarded.

- **ALFRED actuals**: `_fetch_alfred_change` fetches two consecutive vintage prints for `CPIAUCSL`/`PAYEMS` as of the day after `scheduled_time` and diffs them ŌĆö a raw ALFRED level is NOT comparable to ForexFactory's consensus (a period-over-period change), a real bug caught during build. NFP's diff is a raw count (thousands of persons); CPI's is % m/m. `surprise_delta = actual - consensus`, recomputed whenever either value is already present regardless of fetch order.
- **Consensus**: `fetch_and_persist_consensus` matches `event_calendar` against ForexFactory's free `ff_calendar_thisweek.json` feed (title + same-instant match ŌĆö "Non-Farm Employment Change" specifically, not "ADP Non-Farm Employment Change"). Only a "this calendar week" variant exists (nextweek/lastweek both 404) ŌĆö consensus can only ever be captured within roughly ┬▒6 days of "now," a permanent disclosed gap. Fetched at most once per calendar week, persisted to `forexfactory_calendar` (every entry the feed returns, every country ŌĆö keyed `(week_key, title, country, event_date)`) ŌĆö confirmed live that repeat hits trip a 429 and can lock out an IP entirely, so never re-fetched within the same week. This replaced an earlier flat-file cache (`runtime/forexfactory_thisweek.json`) since the file was doing double duty as a second, inconsistent persistence mechanism for data with real long-term value beyond the USD/CPI/NFP subset CATCOR actually matches. `catcor_consensus_actuals`'s `CadenceSpec(trigger="interval", interval_seconds=1800)` in `backend/sources.py` has `_schedule_loop` call it (via `_catcor_consensus_tick`) periodically alongside `fetch_and_persist_actuals`. ForexFactory's `"K"`-suffixed NFP forecasts (e.g. `"114K"`) must be scaled ├Ę1000 to PAYEMS's native thousands unit ŌĆö a second real unit bug caught during build.
- **Snapshot capture** (`catcor.capture_snapshot`): idempotent by construction ŌĆö skips if a `macro_price_reaction` row already exists for `(event_id, metal, window)`. `due_snapshots()` finds boundary-passed-but-uncaptured pairs across **every** `event_calendar` row unconditionally (no `event_type` filter ŌĆö this is what lets Research-promoted `observed` events get their reactions captured automatically, with zero code changes needed for that), polled every 60s by `_schedule_loop` (via `_catcor_snapshot_tick`) under `catcor_snapshot`'s `CadenceSpec(trigger="always_on", interval_seconds=60)` in `backend/sources.py` ŌĆö `always_on` is a distinct trigger value from the ordinary `"interval"` sources specifically so this source is structurally incapable of being gated by an enabled flag, since a missed window is permanent data loss unlike re-fetchable tier data. Price lookup: `db.get_ticks_near` finds the nearest `spot_price_tick` row **under `FUTURES_SERIES_ID`'s keys specifically** (`"XAG_FUTURES"`/`"XAU_FUTURES"`, explicit as of the series-collision fix above ŌĆö CATCOR's reactions were always effectively sourced from Yahoo's futures bars, not real spot ticks, this just makes that explicit rather than accidental); falls back to `backfill_daily_closes`'s dedicated `XAG_DAILY_CLOSE`/`XAU_DAILY_CLOSE` keys (NOT the Money Supply tab's `XAG_CLOSE`/`XAU_CLOSE` keys ŌĆö those are month-end resampled with no daily granularity, a real bug caught during build). `backfill_intraday_ticks` pulls Yahoo 5-minute bars (capped ~72 days back; 1-minute is capped at only 8 days, too short) ŌĆö so ~72 of a 90-day backfill window get real intraday-precision reactions, remaining ~18 oldest days fall back to same-day daily-close approximation.
- **Routes**: `GET /api/catcor/events/db`, `GET /api/catcor/reactions/db`, `POST /api/catcor/refresh` (manual re-seed+backfill+re-fetch, `500` if `FRED_API_KEY` unset).

- ­¤ōī `consensus_value` only populates for events within the current Sun-Sat week ŌĆö events outside it keep `actual_value` but never get `surprise_magnitude`, permanent limitation of the free source, not a bug to chase.
- ­¤ōī Intraday-precision reactions only exist for the ~72 most recent days of the 90-day backfill window.

## Tab: Research (CATCOR Research Pane, per `catcor-events-spec.md`)

**Frontend**: `frontend/src/research_panel.jsx`. **Backend**: `backend/catcor_research.py`, `backend/db.py`'s research tables, `backend/prompts/*.py`.

### Research: what it does

A workbench for working a single claim/observation by hand, one turn at a time, ending in a disposition: promote it to a tracked Catalyst Event (renders on the CATCOR timeline, see below), dismiss it as noise, or discard it outright. Supersedes the original "Iteration 2" chat MVP ŌĆö nothing is auto-fetched by a model deciding it's relevant; every turn is assembled from five independently-adjustable, human-set controls (spec section 3):

1. **Model** ŌĆö `forge` (amp-forge, a local Ollama-backed LAN service) or `anthropic` (Claude), chosen per turn, `forge` is always the default.
2. **Persona** ŌĆö dynamically listed from `backend/prompts/*.py` (see Personas below), no registration step.
3. **Context blocks** ŌĆö six checkboxes (`cot_positioning`, `comex_inventory`, `money_supply`, `market_balance`, `prior_turns`, `freeform`), each folded into the prompt only if explicitly checked.
4. **Memory mode** ŌĆö `stateless` or `accumulating`; a session's current mode is stored and defaults the toggle to wherever it was left.
5. **Prompt transparency** ŌĆö a live, non-editable preview of the exact assembled payload (persona text + checked context blocks' real formatted data + draft input) before sending, computed server-side via `POST /sessions/{id}/preview` (calls `assemble_prompt` directly, zero model cost) rather than approximated client-side; each turn's assembled prompt is also persisted so the transcript can show exactly what was sent, after the fact.

### Session lifecycle

Four states: `active` (editable, only status that can send turns) ŌåÆ `promoted` | `dismissed` (both terminal/read-only) ŌåÆ `discarded` (hard-deleted, never a stored status value ŌĆö `research_sessions`/`research_messages` rows are actually removed, not flagged). Gating rules (`catcor_research.py`):

- **Promote** (`promote_session`) ŌĆö requires a read (bullish/bearish/neutral) already set. Writes a new `event_calendar` row: `event_type="observed"`, `source_tier="discovered"`, `research_session_id` backlinking to the session, `direction` = whatever was chosen in the promote form. `consensus_value`/`actual_value`/`surprise_delta` stay `NULL` (not applicable to an Observed-origin event).
- **Dismiss** (`dismiss_session`) ŌĆö requires a read, a non-empty reason, and Ōēź1 turn already sent (a zero-turn session has nothing to reason about ŌĆö it can only be discarded). Writes a `research_log` row.
- **Discard** (`discard_session`) ŌĆö no gating beyond rejecting an already-terminal session (discarding a promoted/dismissed session would orphan the `event_calendar`/`research_log` row it already produced ŌĆö see Demote below for the actual "undo a promotion" path). Hard-deletes the session and its messages.
- **Demote** (`delete_promoted_event`, new) ŌĆö the reverse of promote: `DELETE /api/catcor/events/{event_id}` deletes the `event_calendar` row and its `macro_price_reaction` rows, then reverts the originating session back to `active` (editable again ŌĆö can be re-promoted with a corrected date/name, dismissed, or discarded). Rejects government-seeded events (`source_tier != "discovered"`) ŌĆö this is only for undoing a Research promotion, never for editing AV's own seeded calendar. Surfaced in the Research tab's **session list** (not the session detail view) as a bulk action: selecting one or more `promoted` rows lights up a Ō¼ć header button (selecting `active` rows instead lights up a ­¤Śæ discard button; a genuinely mixed active+promoted selection leaves both buttons visibly disabled rather than guessing which action was meant).

### Personas (`backend/prompts/*.py`)

The only contract `catcor_research.list_personas()`/`load_persona_prompt()` enforce is a module-level `PROMPT` string constant ŌĆö nothing else in a persona file is read by any code (headers/docstrings are human documentation only, following an established-but-unenforced convention). Personas are listed by scanning `backend/prompts/*.py` (excluding `__init__`/`__pycache__`), so a new file is selectable immediately with no registration step. Live personas:

- `word_count_v1` ŌĆö trivial word-counter, the original plumbing-validation persona, still available but no longer the hardcoded default.
- `parser_v1` ŌĆö the original combined decompose+match persona, built around a `TOOL_REQUEST:` plain-text protocol for a model-driven tool-use loop. **That loop no longer exists** (superseded by the human-checked context-block design above) ŌĆö left in place, unwired, importable, but selecting it will produce inert `TOOL_REQUEST:` lines nothing intercepts.
- `parser_v2` ŌĆö decomposition-only (supersedes `parser_v1`'s combined job, single-responsibility split): claim text in, a numbered list of testable sub-assertions out, each tagged CHECKABLE/NOT CHECKABLE. No tool references, no matching, no verdict.
- `matcher_v1` ŌĆö matching-only, the other half of `parser_v1`'s old combined job: a list of sub-assertions in, each one paired with which of the four AV data types (`cot_positioning`/`comex_inventory`/`money_supply`/`market_balance` ŌĆö same names the context-block checkboxes use) would test it, or `none`. Never fetches or invents real data ŌĆö naming a match is separate from actually reading that data type's current value.
- `parser_v3` ŌĆö currently identical behavior to `parser_v2`; created as a documented template (every available header/option/variable spelled out, since the only real option is `PROMPT` itself) rather than a behavior change.
- `analyst_v1` ŌĆö contextualizing/connecting persona from the old 2-call pipeline (takes already-matched evidence, explains historical-range context, never renders a verdict); still selectable standalone now that any persona can run on any turn.

**No persona, on either backend, has real tool-calling access** ŌĆö `call_anthropic`/`call_forge` are both pure text-in/text-out (no `tools` schema sent to Anthropic's API; amp-forge's payload has no tool-schema field either). This is a deliberate architecture choice, not a Claude-vs-Ollama capability gap: AV data reaches a model exclusively via the context-block mechanism (a human checks a box, Python fetches the real data and pastes formatted text into the prompt **before** the model sees the turn) ŌĆö no persona ever decides what data it gets.

### Model backend layering

`call_anthropic(system_prompt, messages, model)` / `call_forge(same signature)` ŌĆö one function per vendor, identical signature and return shape (`{"final_text": str}`), pure text-in/text-out. `call_ai(backend, ...)` is a thin dispatcher between them.

- **Anthropic**: raw httpx POST to `https://api.anthropic.com/v1/messages` (no SDK, same convention as every other outbound integration in this codebase), model `claude-haiku-4-5-20251001` ŌĆö Haiku specifically, not Sonnet, a deliberate cost choice since this loop is exercised interactively/repeatedly during testing (confirm the exact model string against docs.claude.com if it ever needs correcting). Requires `ANTHROPIC_API_KEY`; has no mock mode ŌĆö a `mock_key` passed to `call_anthropic` raises loudly rather than silently ignoring it, since this is real, billed traffic. `_require_backend_credentials` checks the *per-request* chosen backend, not just the server-wide default, so an explicit `"anthropic"` choice against a `forge`-default server still gets a clean `500` rather than a raw `KeyError`.
- **amp-forge** (`call_forge`): one POST to a local Ollama-backed LAN service's `/chat/stream` (`FORGE_URL`, default `http://amp-forge:8001/chat/stream`; see `forge-spec.md` in the amp-dev repo for the stateless/system-prompt contract), model `qwen3:8b`, accumulates SSE `token` events. Supports a `mock_key` to skip Ollama and stream back a fixed fixture instead. The `history` field in its payload is what powers Accumulating memory mode.
- **`DEFAULT_BACKEND`**, read from `AI_BACKEND` env var, **defaults to `"forge"`** ŌĆö Anthropic is an explicit opt-in for real (billed) compute.

### Context assembly (`assemble_prompt`, `catcor_research.py`)

Four of the six context blocks map to the four evidence-tool functions (`_tool_get_cot_positioning`/`_tool_get_comex_inventory`/`_tool_get_money_supply`/`_tool_get_market_balance`, unchanged from the old pipeline, still also backing `GET /api/catcor/research/evidence/db`'s zero-cost debug dump) via a new dictŌåÆlabeled-text formatting layer (`_fmt_cot_positioning` etc. ŌĆö this formatting step didn't exist anywhere in the codebase before this rebuild). The other two (`prior_turns`, `freeform`) have no backing tool: `prior_turns` controls whether `messages` includes session history at all, `freeform` folds in caller-supplied ad hoc text.

**The one subtle piece**: prior-turn history is included only if `memory_mode == "accumulating"` **AND** `"prior_turns"` is checked ŌĆö both, not either. Stateless always wins regardless of the checkbox (spec 3.4); an unchecked box means that data is absent, full stop (spec 3.3).

### Research: data flow

`POST /api/catcor/research/sessions` ŌåÆ `create_session` (persists session row only) ŌåÆ immediately followed by `send_message` with the claim as the first turn. `send_message` now takes the full five-control signature (`backend`, `model`, `persona`, `context_blocks`, `memory_mode`, `freeform_text`, plus `system_prompt_override`/`messages_override` for the prompt-preview's per-section edit bypass) ŌĆö persists the user turn immediately (with `context_blocks`/`memory_mode`/`memory_changed`/`assembled_prompt` recorded on that row) ŌåÆ persists the assistant turn only if the model call fully resolves (with `backend`/`model`/`persona` recorded on that row). Raises if the session isn't `active` (ŌåÆ route layer returns `409`).

Routes: `POST /sessions`, `GET /sessions/db`, `GET /sessions/{id}/db` (now also returns `promoted_event` ŌĆö an `event_id`/`event_name`/`scheduled_time`/`direction` object whenever `status == "promoted"`, via a new `db.get_event_for_session` reverse lookup), `POST /sessions/{id}/messages`, `POST /sessions/{id}/preview`, `POST /sessions/{id}/read`, `POST /sessions/{id}/promote`, `POST /sessions/{id}/dismiss`, `POST /sessions/{id}/discard`, `DELETE /api/catcor/events/{event_id}` (demote), `GET /personas`, `GET /evidence/db`, `GET /forge-sessions` (stub, see below).

Tables: `research_sessions` (+ `memory_mode` column, tracks the session's current setting). `research_messages` (+ `backend`/`model`/`persona` ŌĆö assistant rows only ŌĆö and `context_blocks`/`memory_mode`/`memory_changed`/`assembled_prompt` ŌĆö user rows only; a "turn" is still a user-row/assistant-row pair, fields split across the two roles rather than duplicated). `research_log` (+ `dismiss_reason`, required non-empty). `event_calendar`'s `research_session_id`/`direction` columns are now actually written (by promote) and read (by demote, and by `list_research_sessions`'s `LEFT JOIN` which surfaces `promoted_event_id` per row for the session list's bulk-demote action) ŌĆö no longer dead columns.

**amp-forge session visibility** (spec 3.4) ŌĆö a known, disclosed gap: amp-forge may hold its own model-side session/context independent of what AV resends, and viewing/clearing that state has no implemented mechanism. `GET /api/catcor/research/forge-sessions` is a stub returning `{"success": false, "detail": "...not yet available..."}` rather than guessing at a wire contract that lives in a separate repo's `forge-spec.md`.

### Research Ōåö CATCOR integration

A promoted session's `event_calendar` row (`event_type="observed"`) renders on the CATCOR timeline exactly like a government-seeded event, with two differences: it's a **circle**, not a diamond (government events keep the diamond shape ŌĆö `makeLinkedShape`'s `diamond` param is now `type !== "observed"`), and its own palette color (`CATCOR_EVENT_COLORS.observed`, amber). Both `get_upcoming_events` and `get_event_reaction_series` (backing `/api/catcor/events/db` and `/api/catcor/reactions/db`) now select `research_session_id`/`direction`, which didn't used to be exposed at all.

Clicking an `observed` point's dot in either CATCOR chart navigates straight to the Research tab and opens that point's originating session (via `App.jsx`'s `openResearchSessionId` state, threaded down to `CatcorPanel`'s `onOpenResearchSession` and `ResearchPanel`'s `openSessionId` prop) ŌĆö a government-seeded point has no `research_session_id` and is a no-op on click. Both chart tooltips show a "Click to open the research record ŌåÆ" hint when a point has one.

The CATCOR legend is now click-to-toggle (same convention as the CoT panel's line legend): clicking any event-type row hides/shows that type on both charts simultaneously (`hiddenTypes` state, filtered before grouping into `pointsByType`/`timelineByType`); a hidden type's legend row dims but the legend itself always shows every type regardless of what's currently plotted.

**Note on reaction capture for promoted events**: `capture_snapshot`'s nearest-tick lookup already worked generically for `observed` events with zero code changes ŌĆö `due_snapshots()` queries `event_calendar` unconditionally, no `event_type` filter ŌĆö so promoting a session with a recent/past `scheduled_time` gets its 4 reaction windows captured automatically on the next 60s poll, same as any CPI/FOMC/NFP event.

## Tab: Data

**Frontend**: `frontend/src/data_editorial.js` (hand-written prose/curl examples, split from the former `data_map.js` per datasources-spec.md Story #1) + `frontend/src/data_panel.jsx`. **Backend**: `backend/sources.py` (canonical operational registry) + `backend/db.py`'s `source_health` table + `main.py`'s `/api/health/*` and `GET /api/data-sources/db` routes.

### Data: what it does

Two things layered together, since datasources-spec.md Story #1's split: a **static, hand-authored editorial map** (`data_editorial.js` ŌĆö provenance prose, curl examples, per-field descriptions; not introspectable from SQLite at runtime, so this must be kept in sync by hand, see the strict Data-tab-update rule under Standing rules above), joined against **operational metadata read live from `backend/sources.py`'s registry** (`GET /api/data-sources/db` ŌĆö cadence, rate limit, affinity group, `requires_env`), plus a **live Data Health view** (per dataHealth-spec.md) showing real-time fetch status per source (`GET /api/health/db`).

### Data: panel contents

`DataPanel` (outer, defaults open) fetches both `/api/health/db` (polled, on refresh) and `/api/data-sources/db` (once per mount ŌĆö operational metadata doesn't change at runtime) ŌåÆ `TieredLoopSummary` (fast/slow tier enabled+interval from `GET /api/refresh/settings`, plus healthy/total rollup per tier from `/api/health/db`'s server-derived `tier` field) ŌåÆ one nested `<details>` per editorial entry (`SourceCard`): origin as a key-value row, then per-`sourceKeys` entry cadence/rate-limit rows (`RateLimitDisplay` ŌĆö Story #6's real enforced state: a live "next eligible fetch: in N days" countdown for `min_gap_derived` sources, the raw quota for `numeric_quota` sources, an explicit "undocumented ŌĆö advisory only" badge for reverse-engineered sources, never a fabricated number) and a `FetchStatusRow` (status dot computed client-side ŌĆö `computeStatus`, exported from `data_panel.jsx` and shared by `App.jsx`'s `HeaderHealthDot` rather than each re-implementing the same ok/stale/error rule ŌĆö against that source's `expected_interval_s`, now shipped directly on each `/api/health/db` row instead of a separate frontend-hardcoded `expectedIntervalS`; last-success time; truncated last-error text; independent "Re-run now" button hitting `POST /api/health/refresh/{source_key}` ŌĆö `cot_pipeline`'s button additionally grays out inside its 7-day rate-limit window), the `curl` block in a horizontally-scrollable `<pre><code>`, then a table per SQLite table listing every field + description.

Sources with no periodic fetch of their own (`catcor_calendar`'s static seed, `research`'s on-demand chat) simply omit `sourceKeys` in `data_editorial.js` (and have no corresponding entry in `backend/sources.py`'s registry).

### Data: data flow

`source_health` (source_key PK, last_attempt_at, last_attempt_status [success|error|skipped], last_success_at, last_error, consecutive_failures ŌĆö upsert, current-state only, not append-only, one row per fetch-and-persist function/loop) is infrastructure describing the health of every other tab's upstream fetch ŌĆö not itself a market-data table, so it has no `data_editorial.js` SourceCard of its own.

**`backend/sources.py`** (per datasources-spec.md Stories #1+#3) is the canonical registry ŌĆö replaces what used to be three separate hand-written dicts (`_FAST_TIER_REGISTRY`/`_SLOW_TIER_REGISTRY`/`_ON_DEMAND_REGISTRY`, merged into `_SOURCE_REGISTRY`) plus a module-level `_SELF_RECORDING_KEYS` set and three bespoke per-source rate-limit `if` blocks. Every source is one `SourceDefinition` (key, label, `affinity_group`, `fetch_fn`, `tables`, `cadence: CadenceSpec`, `rate_limit: RateLimitSpec`, `requires_env`, `self_recording`), registered into `sources.SOURCE_REGISTRY` from `main.py` once every fetch function it references exists as a real name (same ordering constraint the old registries had). `CadenceSpec.trigger` is one of `"interval"` (fired by the generic scheduler on a fixed cadence, subject to `enabled_flag`), `"always_on"` (fired unconditionally, no `enabled_flag` branch ever consulted ŌĆö CATCOR's `catcor_snapshot` reaction-capture source uses this specifically because a missed window is permanent data loss, unlike every re-fetchable source), or `"manual_only"` (reachable only via `POST /api/health/refresh/{source_key}`, plus once at boot if `fire_at_startup=True`). `fire_at_startup` (a separate, orthogonal field, not a fourth trigger value) fires a source once, ~1s after boot, in addition to whatever its `trigger` otherwise does ŌĆö `money_supply`/`metals_prices`/`lbma_fix`/`census_trade` (`trigger="manual_only", fire_at_startup=True`, fire once at boot, never again automatically) and `catcor_startup` (`trigger="interval", fire_at_startup=True, interval_seconds=604800` ŌĆö the former `_catcor_startup` one-shot chain, now also re-running weekly) all use it. A distinct fourth `"startup"` trigger value existed briefly during the initial registry build (dispatched by hand-written `asyncio.create_task(...)` calls in `lifespan`, outside the generic scheduler entirely ŌĆö a third, separate dispatch pattern) and was folded into `fire_at_startup` shortly after, once it became clear that pattern was hiding a real bug: `money_supply`/`metals_prices` were labeled `trigger="startup"` but had no corresponding `lifespan` call, so they never actually fired automatically until this fold fixed it. A single `_schedule_loop()` in `main.py` (ticking every 1s, dispatching per-source on its own `CadenceSpec`) now handles every source uniformly, including the former startup-only ones ŌĆö replacing the old `_fast_tier_loop`/`_slow_tier_loop`/`_event_tier_loop`/`_consensus_tier_loop` four-loops-by-era pattern and the separate `lifespan`-level startup task-creation calls alike. `POST /api/refresh/force` and the one-shot startup refresh still call `_refresh_fast_tier`/`_refresh_slow_tier` directly, now reading `sources.sources_by_tier("fast"/"slow")` instead of the old dicts, so they can't drift from what `_schedule_loop` itself fires.

`GET /api/health/db` remains a thin read of `source_health` plus two fields derived at read time from the matching `SourceDefinition`, never stored: `expected_interval_s` (from `CadenceSpec.expected_interval_s`, the property `2 ├Ś this` staleness threshold is computed against) and `tier` (a small scheduling-only label ŌĆö `"fast"|"slow"|"always-on"|"interval"|"on-demand"`, deliberately distinct from `affinity_group`'s four-value source-category grouping, so `TieredLoopSummary`'s existing fast/slow rollup keeps working unchanged). The route still does not compute ok/stale/error itself ŌĆö the frontend computes status client-side against the shipped `expected_interval_s`.

**Per-source configurable intervals**: `POST /api/data-sources/{source_key}/interval` (`{"interval_seconds": int}`) lets any `trigger="interval"` source's cadence be overridden independently ŌĆö before this, all 11 slow-tier sources shared one `_refresh_settings["slow_interval_s"]` value baked into each `SourceDefinition` once at registry-build time, and mutating that shared setting afterward was a confirmed no-op for already-registered sources (a real pre-existing bug). Overrides live in a mutable `_interval_overrides: dict[str, int]` side-table in `main.py`, checked by `_schedule_loop` *before* falling back to a source's own registered `cadence.interval_seconds` ŌĆö deliberately not a mutation of `CadenceSpec`/`SourceDefinition` themselves, which stay genuinely frozen (their immutability is what makes cadence auditable by construction, per `sources.py`'s own docstring). Persisted in a new `interval_overrides` table (`backend/db.py`, `source_key TEXT PRIMARY KEY, interval_seconds INTEGER`) ŌĆö seeded into `_interval_overrides` once at boot ŌĆö so a deliberate per-source tuning choice survives a restart, unlike the blanket `fast_interval_s`/`slow_interval_s` tier settings, which stay in-memory-only and reset (a considered single-source override was judged more costly to lose silently than the coarser tier-wide toggle). The route rejects `always_on` sources (protects `catcor_snapshot` ŌĆö never overridable into a skippable/slower cadence) and `manual_only` sources (nothing to override ŌĆö no recurring timer exists). `GET /api/data-sources/db` overlays any live override onto its `cadence.interval_seconds`/`expected_interval_s` fields before serializing, so the frontend's existing read picks up an override for free. `frontend/src/data_panel.jsx`'s `SourceCard` renders one `IntervalEditControl` per `sourceKey` (not per card ŌĆö same established per-`sourceKey` granularity its cadence/rate-limit rows already use) inside each `trigger="interval"` row.

A small always-visible header dot (`HeaderHealthDot` in `App.jsx`) polls `/api/health/db` every 60s independent of which tab is active ŌĆö red if any tracked source is erroring, yellow if any is stale with no errors, green otherwise; links nowhere, since the Data tab nav button is already one click away for drill-down.

### Census International Trade (`census_trade`, per international-trade-spec.md)

The one data source that lives in this tab and nowhere else ŌĆö no frontend panel exists for it yet (v1 is backend + DB + Data tab only, per the spec's explicit scope). U.S. Census Bureau International Trade API, monthly imports/exports for HS 7106 (silver) and HS 7108 (gold, comparison-only per the standing "gold as context" rule ŌĆö no gold-specific signal or panel). Answers "where does US silver supply actually come from, and is that mix shifting" with a primary-source view of trade flow (value, quantity, country-of-origin/destination) ŌĆö does not attempt to reproduce USGS's net-import-reliance %, which needs domestic mine-production data AV doesn't track.

`census_trade` table (`metal`, `flow`, `hs_code`, `cty_code`, `cty_name`, `year`, `month`, `value_general_usd`, `value_consumption_usd`, `qty`, `qty_unit`, `fetched_at`; PK `(metal, flow, hs_code, cty_code, year, month)`) ŌĆö **upsert, not append-only** (`INSERT ... ON CONFLICT DO UPDATE`, same shape as `lbma_fix`), since Census revises trade data annually every April, unlike CFTC's immutable-once-published reports.

**Confirmed live** (real `CENSUS_API_KEY`, both flows, both metals, several months, including a real production run): field names are `GEN_VAL_MO`/`CON_VAL_MO` (imports) and `ALL_VAL_MO` (exports) for value, `GEN_QY1_MO`/`CON_QY1_MO` (imports) vs. the differently-named `QTY_1_MO` (exports) for quantity, sharing `UNIT_QY1` for the unit code. **Census reports no quantity/weight at all for HS 7106 or HS 7108** ŌĆö the qty fields are always `"0"` and `UNIT_QY1` is always `"-"` (Census's own not-applicable sentinel), consistently across every row checked. `qty`/`qty_unit` persist as `NULL` accordingly; this is a schema-level upstream gap, not a per-row data hole, and not something to build oz-conversion logic against.

**Also confirmed live, a real correction mid-build**: Census's true publication lag is **~2 months, not ~1** ŌĆö both the current calendar month and the immediately-prior one return HTTP `204` (empty body, not an error status) until released. `_fetch_and_persist_census_trade` treats `204`/empty-body responses as "not yet published, skip this month" and fetches a wider window (`CENSUS_TRADE_MONTHS_PER_FETCH + 2` months back) so 3 real months still land per run. This lag also broke the originally-spec'd rate-limit gate design: gating on "is the latest **persisted** `(year, month)` under 25 days old" (mirroring `_refresh_cot_pipeline`'s CFTC pattern, where report age closely tracks fetch recency) is never true in practice for Census, since by the time a new month is available the previous latest month is already 2+ months old ŌĆö the gate would never actually skip. `_refresh_census_trade` instead gates on **wall-clock time since the last fetch attempt** (`source_health.last_attempt_at`, via a new `db.get_source_health(source_key)` single-row read), which correctly approximates "don't re-check Census more than once per ~25 days" regardless of publication lag.

`_refresh_census_trade` (`main.py`) wraps `_fetch_and_persist_census_trade` with that gate ŌĆö skips (records `last_attempt_status="skipped"`) if the last fetch attempt is under ~25 days old, otherwise fetches and records its own success, since its `SourceDefinition.self_recording=True` (alongside `cot_pipeline`) in `backend/sources.py`. Cadence: `CadenceSpec(trigger="manual_only", fire_at_startup=True, min_gap=timedelta(days=25), gate_on="last_attempt_at")` ŌĆö gated fire-once-at-startup fetch + manual force-refresh via the Data tab's "Re-run now" button ŌĆö silently skips at startup if `CENSUS_API_KEY` isn't set, same as LBMA. `GET /api/census-trade/db?metal=&flow=&hs_code=` is the sole read route ŌĆö no live `/refresh` route, per the same "a government monthly release isn't something a page load should trigger" reasoning as CoT. `GET /api/health/db` enriches `census_trade`'s row with `last_period` (via `db.get_latest_census_trade_period()`), same convention as `cot_pipeline`'s `last_report_date`.

### Data: known gaps

- ­¤ōī `census_trade`'s `qty`/`qty_unit` are confirmed-live always `NULL` ŌĆö Census does not report a quantity/weight figure for HS 7106 or HS 7108 today. Not a bug; revisit only if Census ever starts populating `UNIT_QY1` for these codes.
- ­¤ōī `census_trade` has no frontend panel ŌĆö v1 is backend + DB + this Data tab entry only, per international-trade-spec.md's explicit scope. HS 7112 (scrap) and other Critical Minerals List comparators (e.g. copper 7402/7403) are deferred to a documented Phase 2, not discarded.

This tab's own gaps are otherwise, by design, whatever every other tab's Known gaps sections say about their upstream fetches.

## Conventions

- **Backend**: stdlib `sqlite3` (no ORM), `httpx.AsyncClient` for outbound calls (one shared client via FastAPI lifespan), routes are thin ŌĆö fetch, reshape, upsert, return `{"success": bool, "data": ...}`. HTTP errors from upstream become `HTTPException(502, str(e))`.
- **Pipeline**: intentionally stdlib-only, no third-party deps ŌĆö keep it that way. Config constants belong in `pipeline/config.py`, not scattered inline.
- **Frontend**: React 19 + Vite 5 + Recharts, no state library, no router (single page, tab-like sections via `App.jsx`). Chart colors come from `palette.js`, not ad hoc hex values.
- **Units**: COMEX figures are troy oz; SHFE is kg natively, converted via `1 kg = 32.1507 oz` (constant appears in both `main.py` and any frontend display ŌĆö keep consistent). COMEX open interest contracts are 5,000 oz each.
- **AV Voice Rules** (standing rule, applies repo-wide to any user-facing copy ŌĆö formerly documented in the retired `SPEC.MD`, now stated here as the canonical source): no "alarming"/"critical" framing on deficit trends, always show the ┬▒20% uncertainty qualifier on above-ground stock figures, label runway/track-record figures explicitly as historical description or context ŌĆö never as prediction.
- **Collapsible sections**: the established UI pattern for any collapsible pane is native `<details className="collapsible-pane"><summary className="collapsible-pane-title">ŌĆ”</summary><div className="collapsible-pane-body">ŌĆ”</div></details>` (CSS in `index.css`) ŌĆö nest freely. Prefer this over a `useState` show/hide toggle unless more complex interaction state is needed.
- **Custom legends over Recharts' built-in `<Legend>`**: chart legends in this codebase are hand-rolled (`.comex-legend-list`/`.comex-legend-item`/`.comex-legend-swatch` classes). See `frontend/docs/UI_STANDARDS.md` for the fuller standard this has converged on (horizontal layout, click-not-hover for detail, clicking a legend row must also highlight that series on the chart, always show every relevant item regardless of current value) ŌĆö check new interactive legends/tooltips against that doc rather than inventing a new pattern per chart.
- **Custom scatter-point shapes for cross-chart consistency**: CATCOR's two charts and its legend all use the same diamond shape per event type (`.comex-legend-swatch--diamond` for the legend; a shared `makeLinkedShape` renderer for the charts themselves).

## Running it

```bash
bash utils/vigil.sh start         # everything, as background daemons (venv bootstrap + backend :8000 + frontend :5173)
bash utils/vigil.sh restart backend   # after backend Python edits ŌĆö picks up the change (no --reload running)
bash utils/vigil.sh stop          # stop everything
python3 pipeline/run.py           # CoT pipeline only, no server needed
```

Run the pipeline at least once before the frontend, since the CoT tab reads from `/api/cot/db`, which reads `cot_silver`/`cot_gold` from `runtime/argentvigil.db` ŌĆö empty tables make that route return a `500` ("No CoT data persisted yet. Run pipeline/run.py first.").

The exchange-inventory/FRED/spot-price data populates itself on first backend startup regardless (a one-time unconditional refresh runs in `main.py`'s `lifespan`) ŌĆö the tiered background refresh only *repeats* on a schedule if explicitly enabled via `POST /api/refresh/settings` or forced via `POST /api/refresh/force`. CATCOR's event calendar and reaction backfill also run automatically on every startup ŌĆö no manual trigger needed, though `/api/catcor/refresh` exists for an on-demand re-run. ALFRED calls need `FRED_API_KEY` set in whatever shell launches the backend ŌĆö without it, the event calendar and price reactions still populate, but `actual_value`/`surprise_delta` stay `NULL`. Research's Anthropic backend needs `ANTHROPIC_API_KEY` only if `AI_BACKEND=anthropic` is explicitly set (default is `forge`, which needs no key but does need the `amp-forge` LAN service reachable). The LBMA fix (see Silver/Gold sections above) needs `GAPI_API_KEY` (GoldAPI.io, free tier) ŌĆö without it, `lbma_fix`'s fetch_fn logs a skip message and the rest of the app boots normally; `lbma_fix` just stays empty and `LbmaFixBadge` renders nothing.

**Always run Python through `.venv`, never bare `python3`.** `bash utils/vigil.sh start` creates
`.venv` on first run and installs `requirements.txt` into it (`fastapi`, `uvicorn`, `httpx`,
`python-dotenv` ŌĆö none of these are on system Python). Before running any Python command
against this repo ŌĆö import checks, one-off scripts, `python -c "..."` sanity tests ŌĆö run
`source .venv/bin/activate` first, or invoke `.venv/bin/python` directly. `pipeline/` is the one
exception: it's intentionally stdlib-only by design and *can* run under bare `python3`, but
`main.py`/`db.py`/anything importing `fastapi`, `httpx`, or `dotenv` cannot.

## Coding Modes

Two modes govern how work in this repo is done. **Learning mode is the default** ŌĆö only switch to auto mode when explicitly told to.

### Learning mode (default)

- **Never validate UI changes yourself** ŌĆö no headless browser, no automated screenshotting. The user is at the screen and will look; acknowledge their eyes are better than a headless check for this.
- **Pause after delivering any "demo-able" change** for an interactive walkthrough ŌĆö explain/demo it, be ready to defend the approach or change it. The user always makes the final call.
- **The goal is learning** ŌĆö everything AV touches, from physical vault volumes to how the AI pieces work, not just shipping the change.
- **Ask rather than guess** when something is ambiguous.

### Auto mode (explicit opt-in only)

- **Maximize work done before stopping.**
- **Ask no questions** ŌĆö if a decision is unavoidable, pick the option that would have been recommended anyway.
- **Guess rather than ask.**
